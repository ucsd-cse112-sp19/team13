Basically, this is what I know: So you want less redundant code and more modular HTML code. So you realize that there exists tags like <video> or <iframe> that pretty much have styled/ready-to-use modular html/css/js already builtin. We are basically building that.

The way they achieve this is by using templates, the ShadowDOM, and custom tags.

Note: If you are familiar with React, this is very similar to how React components are used. The main difference (besides setup code, how it works, VirtualDOM, etc.) is that React also controls data between/within components with props and states. Web components have nothing to do with managing data. This makes Web components less cumbersome and more flexible (and supported everywhere). This also means it is up to us to make a safe, consistent “data flow” between our components (something like Redux). Just something to keep in mind.

TL;DR

Use <template> to create your html DOM subtree with <style> and <script> (no external files). <template> tags will be “inert” (meaning stuff inside does not execute on load). This subtree is also completely isolated from other code, so styling and logic cannot cross the SHADOW BOUNDARY! By itself, this does nothing. Then register it with the ShadowDOM to make it actually mean something and make it a custom-tag so it’s not just div tags everywhere (they are meaningless, no semantic value). There are ShadowDOM events you can listen to, like created, etc. You can use the <content select=””> tag to specify parameters for the user. Now you can use it like:

<template>
	<content select=”img”>
	...
</template>
<script>
	//Registration stuff...
</script>
<i-am-your-tag>
	<img … />
	<img … />
	<img … />
</i-am-your-tag>

Templates

For web components, templates are how you specify the layout of that inner DOM. It’s basically HTML/CSS(cause you MUST you style tags)/JS that represent the thing you want wrapped in <template>. However, everything inside these tags are NOT executed immediately on load, only when needed. So script tags, etc. won’t trigger until later.

ShadowDOM

The ShadowDOM is a separate place completely disconnected from the one true DOM (meaning that styles, javascript, etc cannot reach out or in the ShadowDOM subtree or the DOM. i.e. styling h1 outside, does not affect inside). You can have multiple ShadowDOM’s, one for each custom-tag you want.

Here’s how you create a ShadowRoot (basically the container for your shadow content).

<div class="container"></div>
	<script>
var host = document.querySelector('.container');
var root = host.createShadowRoot();
root.innerHTML = '<p>How <em>you</em> doin?</p>'
</script>

Now, attaching the template to the ShadowDOM

<template>
<!-- Full of slider awesomeness -->
</template>

<div class="img-slider"></div>

<script>
// Add the template to the Shadow DOM
var tmpl = document.querySelector('template');
var host = document.querySelector('.img-slider');
var root = host.createShadowRoot();
root.appendChild(document.importNode(tmpl.content, true));
</script>

Now to allow something like:

<video>
		<source> ...
	</video>

You can use a <content> tag that selects these child tags, like parameters. They are used in <template> blocks. You can then style them in the ShadowDOM’s css with ::content and other things.

<template>
…
<div class="inner">
<content select="img"></content>
</div>
</template>

Now we are not done yet. Currently, you can use it like this, but it isn’t as pretty as it could be. For example:

<div class="img-slider">
  <img src="images/rock.jpg" alt="an interesting rock">
  <img src="images/grooves.jpg" alt="some neat grooves">
  <img src="images/arch.jpg" alt="a rock arch">
  <img src="images/sunset.jpg" alt="a dramatic sunset">
</div>

This still makes us div-happy. So here we use custom-tags to be more specific. This is done through code.

Here’s the code that the tutorial I read had:

<template> … </template>
<script>
  // Grab our template full of slider markup and styles
  var tmpl = document.querySelector('template');

  // Create a prototype for a new element that extends HTMLElement
  var ImgSliderProto = Object.create(HTMLElement.prototype);

  // Setup our Shadow DOM and clone the template
  ImgSliderProto.createdCallback = function() {
    var root = this.createShadowRoot();
    root.appendChild(document.importNode(tmpl.content, true));
  };

  // Register our new element
  var ImgSlider = document.registerElement('img-slider', {
    prototype: ImgSliderProto
  });
</script>

Here’s the way I think is better:

<template> … </template>
<!-- They don’t have to be style tags, you can use external files too -->
	<script>
const template = document.querySelector('template');
class ImgSlider extends HTMLElement
{
	constructor() { super(); }
	createdCallback()
{
	Let root = this.createShadowRoot();
	root.appendChild(document.importNode(template.content, true));
}
}
document.registerElement(‘img-slider, ImgSlider);
</script>

Since this is using ES6, we should use Babel or something to make it compatible with older browsers. In addition, I think we should use a uglify/minify thing and some sort of dev-server (hopefully with hot-reload of some kind to speed things up). We could also use some sort of bundler if we want, like webpack or something, but they do get messy really quick, so I would suggest caution if we are going down that route (I’m currently in a project using webpack, and although it automates a lot of things, it is challenging/time-consuming to update/change.)
