Question 1) E2E testing in an application like Amazon is simple to think about: start from user logging in to user browsing and buying something to user logging out. But how does E2E testing work for web components?

A:

TL;DR

  • We should not be writing anymore tests for E2E but rather we should just be calling a sequence of unit tests while specifying a different runner.
  • A unit test should have only one assertion

First consider what is meant by application. In terms of Amazon, there will be a user interface that user’s interact with. Then there is data flow from parts of the application which changes the state. So an app contains UI, data flow, and state service.

Now in terms of web components. Say we have an input field and a slider. The value of the slider will determine the font-size of the text in the input field.

Before writing an E2E test, we need to first draw an activity diagram:

Once we have an activity diagram, we can begin writing an E2E test. Before that, we need to see how to write tests in general. For web components, there are only a few things to test and those are key-value pairs (ex: <input font-size=10 />). To test key-value pair font-size, we need two tests, testFontSizeGet() and testFontSizeSet(). We’ll show how to structure the test for testFontSizeSet() since the other will be exactly the same:

testFontSizeSet()

Right now, this diagram screams unit test. The unit tests for input font-size and slider would have a runner (in our case Mocha is our Test Runner) which sets up and cleans up the environment before and after the test runs. Now to make an E2E test, we use the same unit tests for input font-size and slider except we use a different runner, call it InputSliderInteractionRunner, instead of the runners associated with the unit tests. This way, the slider and input share the same environment throughout the E2E test.

Question 3) Please provide tips on writing unit tests for components. For instance a slider can have a vertical attribute which determines it orientation and we can programmatically check that the attribute has been set but we don’t know if setting the attribute resulted in the desired behavior visually.

A:

TL;DR

  • Checking whether setting the vertical attribute oriented the slider vertically in the browser is a UI test NOT a unit test.
  • UI tests are for testing visual aspects and unit tests are for programmatic aspects

Checking whether setting the vertical attribute oriented the slider vertically in the browser is a UI test. As the Professor demonstrated in class, we can use the tool Percy.io to create UI tests. Also, please do not confuse the types of tests going on:

  1. setVertical(true) 2) Visual inspection

1) Is a unit test where we need to see whether calling setVertical(true) actually sets the attribute vertical to true (i.e <core-slider vertical=true> </core-slider> happens)

2) Is a UI test where we see that the slider is visually vertical regardless of what attribute vertical is set to because that’s handled by 1)

Question 4) Should the whole team be writing unit tests or just a small portion?

A:

TL;DR

  • Developers writing the methods need to write getters and setters for the attribute/functionality they are implementing (ex for slider: getVertical(), setVertical(boolean))
  • QA need to write the assertions (i.e. test cases)

Let’s say a developer is implementing the vertical functionality for a slider. Then the developer would need to provide getters and setters getVertical() and setVertical(boolean). These methods are for programmatic use in testing. These are completely separate from what the user will do. End users won’t call getVertical() and setVertical(boolean) but will just use the standard html like

<core-slider vertical=true></core-slider>
