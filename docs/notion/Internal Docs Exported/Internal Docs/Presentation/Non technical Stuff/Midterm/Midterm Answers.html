  1. Definitions
    1. In your own words try to precisely define Software Engineering. (Hint: Focus on the second word)
Software engineering is the production of software through a series of well-defined processes with an emphasis on reliability, security, maintainability, and repeatability. Software engineering is not a “miraculous event”, but the result of a strict adherence to processes.Software engineering is an application of a widely researched and accepted methodology in designing, developing and maintaining software in order to reach the highest possible effectiveness.Just like engineering processes, we need to make our software development more repeatable and ‘engineering-like’ even though we may think of it as ‘artistry’.   1. Define what you think Code Craftsmanship involves
Code craftsmanship describes the artistic processes behind software engineering; craftsmanship involves creativity, ingenuity, and out-of-the-box thinking. It is the more emotional element behind software.
  1. Define what you think Software Manufacturing involves?
Software manufacturing is the process of producing software in ways similar to the manufacturing of tangible goods. It is methodical, repeatable, and largely automated. It describes the more logical, less people-involved element behind software. Hence an important part of this process is the conveyor belt which allows code to go through the same processes to produce predictable results.
  1. Does Software Engineering contain these ideas or is it separate and different? Explain your opinion
Software engineering contains these ideas; without the craftsmanship, there is no ingenuity in designing software solutions to problems; we need the human element to come up with solutions in the first place. However, without the manufacturing part, our process of building these solutions devolves into chaos. Software that works becomes a “miracle event”, and there is no repeatable and rational way to build software solutions.

    1. The Google team study and the HRT methodology emphasis a number of dynamics and characteristics that lead to high performing teams. Describe these characteristics
Google Study Reveals the following primary characteristics among high performing teams:
  1. Psychological Safety - Team members feel that their position and standing in the team is not jeopardized by failure; they trust their teammates to support them and not to judge them.
  1. Dependability - Team members can trust their fellow teammates word; all members of the team follow through on their promises; no one is habitually late to meetings, doesn’t do their work, etc.
  1. Structure and Clarity - Expectations and duties are made clear to all team members; regularly scheduled team meetings establish goals in the short and long term; open channels of communication enable clarification whenever necessary
  1. Meaning of Work - Team members understand where their work fits into the greater goal of the team; each member sees themselves as part of the whole and each feels that everyone’s work is a vital piece of the puzzle.
  1. Impact of Work - Team members understand the greater goal of the team and consider it a worthwhile approach. Regardless of the size of the project, the team sees it as an important endeavor.
HRT (Heart) breaks down core team dynamics into 3 characteristics:
  1. Humility - No big egos or people who consider themselves above the rest. Everyone is willing and able to take feedback and criticisms to heart.
  1. Respect - Team members show appreciation for each other, and treat each other as human beings on equal footing.
  1. Trust - Knowing that you team members are capable and will do their work without your intervention

  • TDD is the process of writing initially failing test cases that define a desired improvement and then writing the minimum amount of code needed to pass the tests.

  1. Code Reviews can be quite difficult to perform. Explain why you should do them. Next explain what the biggest dangers badly run code reviews may lead to. Finally explain at least three tips on performing good code reviews.
Why should we do code review?
  • Help to improve code quality and maintainability
  • Help to find defects in code to ensure correctness, and identify potential performance and security issues in the code
  • Help to share knowledge across the teams, it help all team members better understand the project
  • Help to increase the sense of collective code ownership and responsibility for team
  • Help to find potential better solution for the problem
Biggest dangers badly run code review may lead to?
  • Code bashing : make fun of the code the programmer wrote
  • Create divisiveness among team members; teammates view some as “weak links”
  • Decreased productivity, developers don’t want to write code that will get bashed
Tips on performing good code review
  • Has a clear goal of the code review, and has a standard that the teams agree on the quality of code
  • Keep each code review section short, just review one or two functionality to avoid confusion and move on when appropriate
  • Not just try to identify the problem, but finding a way to solve the problem.
  • Keeping documentation / note on the code review, so it can be referred on future development
  • Emphasize learning from failure, not the failure itself
  • Treat all code as the team’s code, not each individual programmer’s code
  • Review small segments of code to focus on quality, not quantity.
  • Be critical, but also empathetic and understanding.
  • Communicate with every member.
