TestCafe Handguide

Important stuff required for using Test Cafe

What is TestCafe

TestCafe allows you to write tests using TypeScript or JavaScript (with its modern features like async/await). You get all the advantages of strongly-typed languages like rich coding assistance, painless scalability, check-as-you-type code verification, etc., by using TypeScript to write your TestCafe tests.

This definition is based on https://devexpress.github.io/testcafe/documentation/getting-started/

https://devexpress.github.io/testcafe/documentation/test-api/assertions/

Required Files

  • TestCafe JS testing file
  • Testing HTML page and its dependency

Testing JS File

Setting up TestCafe is easy, you just need to add an empty JS file and add the following code:

import { Selector } from 'testcafe';

A selector is an important part of TestCafe. It is used to select an element from the HTML page. It works just like document.querySelector(). More info will be provided later below.

/* eslint-disable */
fixture `Core component Test`
    .page `./test.html`;
/* eslint-enable */

This code block gets the HTML page that will be tested. Make sure to make it as close as possible to the finished product

Testing HTML Page

This is the page indicated by the fixture which by the example above would be "test.html". An example of such a page is shown below

<!DOCTYPE html>
<html>
    <head>
        <title>Testing for core-slider</title>
        <script src="../dist/core-slider.js"></script>
    </head>
    <body>
        <core-slider id="mySlider"></core-slider>
    </body>
</html>

This is the core of what a test page needs. The title tag should indicate what the document is for; in this case, we know that it is for testing the core-slider. One of the most important items to include is the script tag which must link to the js file of the component which is usually found in the "dist" folder. Inside the body is where you can create the components to manipulate; be sure to give each instantiated component unique ids for unambiguous referencing.

Testing Conventions and Rules

Here's a list of guidelines to follow when writing tests

  • All tests are contained in the "test" directory
  • All unit tests for a single component are contained inside a single js file
  • The js file containing the unit tests must be named "nameOfComponent.testcafe.js"
  • The html documents used to test components must be in the "test" directory

Selector Syntax

The common method to use the Selector is by using the listed format below

// Selecting one element
const testElement = Selector('#ElementId')

// More detailed selection
const testElementShadow = 
Selector(() => document.querySelector('#ElementId')
											 .shadowRoot.querySelector('#ElementIdShadow'));

Testing Syntax

The basic syntax for each unit test is as described as below

/**
 * What does it do tho

		const NameInput = Selector('#myInputBox') // Get the element you are testing
  await t
    //.doSomething
		.typeText(NameInput, 'Peter') // insert values into field
    .expect(element.value).eql('value'); // check for equality between arbitrary values
		.expect(Selector("#element_id").visible).ok(); // check that value equals 'true'

Starting the Test

Starting the test is easy, you need the following command in your terminal

testcafe chrome test/test.testcafe.js

Alternatively, you can use the following command. Just check package.json to see what it does

npm test

If you want to do the test without creating an instance of chrome, you can add :headless

testcafe chrome:headless test/test.testcafe.js

There is many choice for browser test, you can use the following browser

  • Google Chrome: Stable, Beta, Dev and Canary
  • Internet Explorer (11+)
  • Microsoft Edge
  • Mozilla Firefox
